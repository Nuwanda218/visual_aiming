# visual-assisted-aiming 项目阅读说明

这份 README 的目标不是教你“怎么立刻改代码”，而是先帮你建立一张完整的脑图：

1. 这个程序到底在做什么
2. 它由哪些模块组成
3. 一帧图像是怎么一步一步变成鼠标移动的
4. 哪些变量是核心状态
5. 你应该按什么顺序读代码

如果你能把这几件事看明白，后面继续优化识别、瞄点和压枪，就不会只是在“碰运气改参数”。

---

## 1. 项目总体目标

这个项目本质上是一个**实时闭环控制程序**。

它做的事情可以概括成一句话：

> 截图 -> 识别目标 -> 算出瞄准点 -> 算出鼠标应该怎么移动 -> 发送鼠标移动

其中，主循环不断重复这个过程。

如果再细一点，可以拆成下面 5 步：

1. 获取当前是否处于辅助激活状态
2. 从屏幕中央固定区域截图
3. 在截图中检测目标轮廓
4. 根据轮廓计算一个“应该锁定的位置”
5. 把这个位置转换成鼠标相对移动

如果左键正在长按，还会额外叠加一层压枪补偿。

---

## 2. 先看程序主流程

整个程序的调度中心在 `main.py`。

你可以把它理解成“总导演”，其他模块都只是被它调用。

主流程大致是这样的：

1. 检查管理员权限
2. 读取 `config.json`
3. 计算屏幕中心和固定截图区域 ROI
4. 初始化所有功能模块
5. 进入 `while not wakeup.should_exit():` 主循环
6. 每一轮循环里：
   - 判断辅助是否激活
   - 判断左键是否按下，决定是否进入压枪状态
   - 根据节流器决定这一帧要不要做识别
   - 截图
   - 检测目标
   - 计算瞄准点
   - 叠加压枪偏移
   - 调用鼠标控制模块移动鼠标

所以你阅读这类项目时，最重要的思路是：

> 不是从某个算法文件开始看，而是先从 `main.py` 看“谁在调用谁”。

## 3. 模块总览

下面是整个项目的核心模块。

### `main.py`

职责：总控制器。

它不负责具体算法，只负责把各模块串起来。

它做的几件事最重要：

- 初始化配置和屏幕中心
- 创建所有模块对象
- 维护主循环
- 在循环里调用截图、检测、瞄点计算、压枪和鼠标移动

可以把它理解为：

> “业务流程层”

---

### `config.py` + `config.json`

职责：配置管理。

`config.py` 里定义 `Config` 类，提供默认参数；`config.json` 是实际运行时读取的外部配置文件。

这里要特别理解一个关键点：

- `config.py` 定义的是“程序认识哪些配置项”
- `config.json` 提供的是“运行时想给这些配置项什么值”

当前加载逻辑是：

```python
for k, v in data.items():
    if hasattr(self, k):
        setattr(self, k, v)
```

这意味着：

> 只有 `Config` 类里已经定义过的字段，`config.json` 里的值才会真正生效。

所以如果以后你在 `config.json` 里加了新参数，但 `Config` 类里没有对应字段，那么这个新参数实际上会被忽略。

这是这个项目里一个非常容易忽略的点。

---

### `input_listener.py`

职责：监听键盘鼠标输入，决定辅助何时开启、何时关闭。

核心类：`WakeUpModule`

它内部维护了很多“状态变量”：

- `shift_pressed`
- `right_pressed`
- `left_held`
- `active`
- `exit_flag`
- `calibrated_crosshair`
- `fixed_roi_offset`

你可以把它理解成：

> “程序的输入状态管理器”

这个模块不算瞄点，不截图，也不移动鼠标，它只负责告诉主循环：

- 现在辅助有没有激活
- 左键是不是按住了
- 默认准星在哪里
- 程序是否应该退出

激活逻辑大致是：

- 按住 `Shift`
- 按住右键
- 再按下左键
- 程序进入 `active = True`

---

### `screen_capture.py`

职责：截图。

核心类：`ScreenCapture`

它每次只截取一个固定的屏幕中心区域 ROI，而不是全屏。

这件事很重要，因为：

- 截图区域越小，处理越快
- 目标通常就在准星附近，没必要全屏检测

这个模块的输出非常简单：

- 输入：ROI 的左上角坐标、宽、高
- 输出：一张 `numpy` 数组形式的图像

你可以把它理解成：

> “图像输入模块”

---

### `detection.py`

职责：在截图里找到目标。

核心类：`TargetDetector`

当前的识别思路不是深度学习，也不是模板匹配，而是：

1. 拿到截图
2. 分离 `BGR/RGB` 通道
3. 用固定颜色阈值做 `mask`
4. 剔除绿色标记
5. 做形态学处理
6. 查找轮廓
7. 筛掉面积和宽高比不合适的轮廓
8. 选出“最可能是目标”的那个轮廓

这个模块输出的是一个 `DetectedTarget` 对象，里面最重要的是：

- `contour`：轮廓点集
- `bbox`：外接矩形
- `center_x / center_y`
- `aspect_ratio`
- `area`

所以 detection 模块的本质是：

> “把原始图像变成结构化目标信息”

注意：

它并不直接告诉鼠标怎么动，它只负责说：

> “我认为目标在这里，这个轮廓长这样。”

---

### `aim_calculator.py`

职责：把“目标轮廓”变成“瞄准点坐标”。

核心类：`AimPointCalculator`

这是整个项目里最像“算法核心”的模块。

它的输入是：

- `DetectedTarget`
- ROI 左上角坐标

它的输出是：

- 屏幕上的一个绝对坐标 `(x, y)`

当前逻辑大致是：

1. 从目标轮廓得到凸包
2. 在凸包内部搜索一个较大的矩形
3. 取这个矩形中靠上方的位置作为瞄准点
4. 如果失败，再退回到外接框位置
5. 再失败就退回到质心

它还额外做了两件事：

- 非开火状态下，对瞄点做平滑
- 开火状态下，维护一个“锁定点”并进行有限跟随

所以这个模块可以理解为：

> “从目标几何信息中推导出真正想锁的位置”

你之前提到的“凸包内最大矩形”优化，就是发生在这个模块里。

这也是当前项目最容易拖慢性能的地方，因为这里有较多 Python 层循环和 `pointPolygonTest`。

---

### `recoil.py`

职责：压枪补偿。

核心类：`RecoilCompensator`

它会从 `recoil_profile.json` 读取一条后坐力曲线。

曲线里每个点大概表示：

- 时间 `t`
- 枪口累计偏移 `dx`
- 枪口累计偏移 `dy`

程序在开火时会记录 `firing_start_time`，然后根据当前已经开火多久，在曲线上插值出当前的补偿量。

这个模块本质是：

> “给主循环提供一个随时间变化的压枪偏移”

它并不管识别，也不管瞄点，只负责回答：

> “现在应该额外往哪个方向压多少像素？”

---

### `mouse_control.py`

职责：把“目标绝对坐标”变成“本帧鼠标相对移动”。

核心类：`MouseController`

这个模块做的不是“瞬移到目标”，而是：

1. 读取当前鼠标位置
2. 计算当前鼠标到目标点的差值 `dx, dy`
3. 判断是否进入死区
4. 限制单帧最大步长
5. 加一点噪声/抖动
6. 用 Windows API 发送相对移动

所以这个模块本质上是：

> “运动控制器”

它不关心目标怎么识别出来，也不关心压枪曲线怎么来的，它只负责：

> “现在我应该让鼠标往哪里挪一点点”

---

### `throttle.py`

职责：节流。

核心类：`Throttle`

这个模块的目的不是“提高识别准确率”，而是“限制辅助触发频率”。

当前有两种限制方式：

- 概率跳过：按 `adsorb_prob`
- 时间片限制：通过 token 方式限制单位时间内可用时长

如果 `allow()` 返回 `False`，主循环这一帧就不会做完整的检测和吸附逻辑。

所以它本质是：

> “主循环是否执行识别的开关”

---

### `debug_visualizer.py`

职责：调试可视化。

它会把以下信息画到窗口里：

- 原始轮廓
- 外接框
- 凸包
- 估计矩形
- 瞄准点
- 校准点

这个模块的意义是帮助你“看见程序脑子里在想什么”。

但是它要注意两个问题：

1. 它本身也会消耗性能
2. 它里面为了画图，有些几何计算会重复做一次

所以你应该把它理解为：

> “观察程序内部状态的工具”，而不是核心功能本身

---

### `utils.py`

职责：通用小工具。

当前只有一个 `ThrottledPrinter`，用来限制日志输出频率，避免某些信息每一帧都刷屏。

---

### `build.py`

职责：打包脚本。

它和识别逻辑没有直接关系，只负责：

1. 清理旧的 `build/` 和 `dist/`
2. 调用 `pyinstaller`
3. 打包成 `exe`
4. 复制配置文件到 `dist/`

所以它属于：

> “工程辅助脚本”

---

## 4. 一帧数据是怎么流动的

你可以把程序每次循环想成下面这个流程：

### 第一步：判断程序是否工作

主循环先问 `WakeUpModule`：

- 辅助有没有激活？
- 左键是否按住？
- 是否应该退出？

如果没激活，很多后续逻辑都不会执行。

---

### 第二步：截图

`ScreenCapture.grab()` 截取屏幕中心 ROI。

输出：

- 一张 `frame`

---

### 第三步：目标检测

`TargetDetector.detect(frame, config, roi_center)`

输出：

- `DetectedTarget`
- 或者 `None`

如果这里返回 `None`，说明“这一帧没有识别出有效目标”。

---

### 第四步：瞄准点计算

`AimPointCalculator.calculate(target, roi_left, roi_top)`

输出：

- 一个屏幕绝对坐标 `(x, y)`
- 或者 `None`

如果这里返回 `None`，说明“虽然可能有目标轮廓，但没能成功推导出瞄准点”。

---

### 第五步：压枪补偿

如果当前正在开火：

`RecoilCompensator.get_current_offset(time.time())`

输出：

- `(comp_x, comp_y)`

然后把补偿量叠加到瞄准点上。

---

### 第六步：鼠标移动

`MouseController.move_towards(target)`

这里的 `target` 已经是：

- 瞄准点
- 加上压枪补偿

鼠标控制模块只负责朝这个目标一点点移动。

---

## 5. 这个项目里最重要的状态变量

如果你是初学者，建议优先抓住下面这些变量，因为它们是真正控制程序行为的核心。

### `WakeUpModule` 中的状态

- `active`
- `left_held`
- `right_pressed`
- `shift_pressed`
- `calibrated_crosshair`
- `fixed_roi_offset`

它们决定程序是否工作，以及“默认准星”在哪里。

### `main.py` 中的状态

- `firing`
- `firing_start_time`
- `last_aim_base`

它们决定：

- 当前是否进入压枪状态
- 压枪曲线从什么时候开始算
- 当某一帧失败时，程序是否还能沿用上一帧的目标点

### `AimPointCalculator` 中的状态

- `locked_aim`
- `_prev_aim_global`
- `_velocity`
- `track_lost_frames`

它们决定：

- 当前是不是“锁定某个瞄点”
- 是否进行平滑
- 丢目标之后能坚持几帧

### `TargetDetector` 中的状态

- `last_target_center`
- `last_target_bbox`
- `track_lost_frames`

它们决定：

- 当前是否在追踪“上一帧那个目标”
- 还是每一帧都重新找一个新目标

## 6. 这个项目最容易让初学者混淆的地方

### 混淆点 1：检测目标 != 算出瞄准点

很多时候你会以为“检测到了目标，就应该能移动鼠标”。

实际上这是两个阶段：

1. `detection.py` 找到目标轮廓
2. `aim_calculator.py` 从轮廓算出瞄准点

所以：

- `detect()` 成功，不代表一定能移动
- 还要看 `calculate()` 是否成功

---

### 混淆点 2：瞄准点 != 最终鼠标目标

最终鼠标移动的目标并不是“纯瞄准点”。

还可能叠加：

- 压枪补偿
- 节流效果
- 死区限制
- 单帧最大步长限制
- 抖动和噪声

所以鼠标表现出来的结果，往往是多个模块叠加的综合效果。

---

### 混淆点 3：`config.json` 不是万能的

`config.json` 里的参数，并不保证一定被程序使用。

只有当：

1. `Config` 类中定义了这个字段
2. 某个模块真的读取了这个字段

这个参数才算真正生效。

所以以后如果你改了 `config.json` 却发现程序没变化，第一件事要检查：

> 这个配置项到底有没有被代码真正读取？

---

### 混淆点 4：回到屏幕中心不一定是鼠标模块的问题

如果程序某一帧没有拿到有效目标点，它可能会走回退逻辑。

这时鼠标看起来像“吸回屏幕中心”，但真正原因往往是：

- 检测失败
- 瞄点计算失败
- 回退逻辑选择了默认准星

所以你调试这类问题时，要沿着链路往前追，不要只看鼠标控制模块。

## 7. 如果你要开始优化，应该先按什么顺序理解

非常建议你按下面顺序读代码：

### 第 1 步：读 `main.py`

目标：

- 理解主循环每一轮做了什么
- 看清楚模块调用顺序
- 搞明白输入和输出在哪里连接

### 第 2 步：读 `input_listener.py`

目标：

- 理解辅助何时激活
- 理解 `active`、`left_held` 是怎么来的

### 第 3 步：读 `screen_capture.py`

目标：

- 理解 frame 从哪里来

### 第 4 步：读 `detection.py`

目标：

- 理解目标是怎么从图像里被提取出来的
- 理解 `mask -> contour -> target` 这条链

### 第 5 步：读 `aim_calculator.py`

目标：

- 理解“为什么目标轮廓会变成一个瞄准点”
- 理解锁定、平滑、跟随的区别

### 第 6 步：读 `recoil.py` 和 `mouse_control.py`

目标：

- 理解最终鼠标移动是怎么形成的
- 理解压枪补偿和瞄准点是如何叠加的

### 第 7 步：最后再看 `debug_visualizer.py` 和 `build.py`

目标：

- 把辅助工具和核心逻辑区分开

## 8. 如果你现在就想开始做优化，最值得先看的三块

### 第一块：`detection.py`

因为它决定“有没有目标”。

如果这里经常输出 `None`，后面所有逻辑都会失效。

### 第二块：`aim_calculator.py`

因为它决定“目标轮廓要瞄哪里”，而且这是目前最重的计算之一。

### 第三块：`main.py`

因为很多“看起来像算法问题”的现象，实际上都是主循环里的状态和回退逻辑造成的。

## 9. 你现在可以怎么学这个项目

如果你是计算机专业新手，我建议你不要一上来就优化算法，而是先做下面三件事：

1. 手写一张纸上的模块图
   - `main -> input -> capture -> detection -> aim -> recoil -> mouse`
2. 选一帧数据，自己口头复述它是怎么流动的
3. 对每个模块回答一个问题：
   - 输入是什么？
   - 输出是什么？
   - 它维护了哪些内部状态？

只要你能把这三件事说清楚，你就已经不再是“只会提需求的人”了，而是在真正理解这份代码。
